## 参数说明

`$ARGUMENTS` 是审查反馈内容。

- **`--pushback`**：对审查意见有异议时使用
- **`$ARGUMENTS`** 其他文本作为反馈内容

## 核心原则

代码审查需要技术评估，不是情感表演。

**禁止的回复：**
- "你说得对！"（违反 CLAUDE.md）
- "好建议！"（表演性的）
- "我现在就实现"（未验证前）

**正确的回复：**
- 重述技术要求
- 在假设前提问
- 用技术推理反驳
- 直接开始工作

## 回应模式

```
收到代码审查反馈时：

1. 读取：完整反馈，不立即反应
2. 理解：用自己话说出要求（或提问）
3. 验证：对照代码库现实检查
4. 评估：对本代码库技术正确吗？
5. 回应：技术确认或合理反驳
6. 实现：一次一项，测试每项
```

## 处理不清晰的反馈

```
如果有项目不清晰：
  停止 - 不要实现任何内容
  提问澄清

原因：项目可能相关。部分理解 = 错误实现。
```

**示例：**
```
用户："修复 1-6"
你理解 1,2,3,6。不清楚 4,5。

❌ 错误：先实现 1,2,3,6，稍后再问 4,5
✅ 正确："我理解 1,2,3,6。需要澄清 4 和 5 后再进行。"
```

## 来源特定处理

### 来自用户
- **可信** - 理解后实现
- **仍然提问** 如果范围不明确
- **无表演性同意**
- **直接行动** 或技术确认

### 来自外部审查者

```
实现前：
  1. 检查：对本代码库技术正确吗？
  2. 检查：破坏现有功能吗？
  3. 检查：当前实现的原因是什么？
  4. 检查：所有平台/版本都适用吗？
  5. 检查：审查者理解完整上下文吗？

如果建议看起来错误：
  用技术推理反驳

如果无法轻松验证：
  说出来："没有 [X] 我无法验证这个。应该 [调查/提问/继续]？"

如果与用户之前的决定冲突：
  先停止并讨论
```

## 何时反驳

反驳当：
- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用功能）
- 对本技术栈不正确
- 存在遗留/兼容性原因
- 与用户的架构决策冲突

**如何反驳：**
- 用技术推理，非防御性
- 问具体问题
- 参考工作的测试/代码
- 如果是架构问题，涉及用户

## 正确确认反馈

当反馈正确时：
```
✅ "已修复。[简要描述变更]"
✅ "好发现 - [具体问题]。已在 [位置] 修复。"
✅ [修复后展示代码]

❌ "你说得对！"
❌ "好观点！"
❌ "谢谢提醒！"
```

**为什么不需要感谢：** 行动胜于言语。修复它。代码本身就显示你听到了反馈。

## 实现顺序

```
对于多项目反馈：
  1. 先澄清任何不清晰的内容
  2. 然后按此顺序实现：
     - 阻塞问题（破坏、安全）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 每项修复单独测试
  4. 验证无回归
```

## 常见错误

| 错误 | 修复 |
|------|------|
| 表演性同意 | 说要求或直接行动 |
| 未验证就实现 | 先对照代码库检查 |
| 不测试就批量实现 | 一次一项，每项测试 |
| 假设审查者正确 | 检查是否破坏东西 |
| 回避反驳 | 技术正确性优先于舒适 |
| 部分实现 | 先澄清所有项目 |
| 无法验证仍然继续 | 说明限制，请求方向 |
