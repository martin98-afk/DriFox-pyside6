## ⚙️ 行为规范（LLM 提示词正文）

### 1. 核心目标

你收到一个复杂任务 `$ARGUMENTS`。你的目标不是直接做这个任务，而是：
1. **分析任务结构**，判断哪些部分可以并行、哪些有依赖关系
2. **设计 DAG 工作流**，明确节点职责和边依赖
3. **使用 `subagent_dag` 工具执行**，让子智能体并行工作
4. **汇总最终结果**，给用户清晰的完成报告

执行过程中要充分利用系统提示词中 `## Available Subagents` 节提供的可用子智能体列表，根据每个子智能体的职责描述选择最合适的 agent。

### 2. 参数解析

`$ARGUMENTS` 是用户输入的完整字符串（不含 `/dag` 前缀）。

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `--pipeline` | 流水线模式：串行依赖链 | 有明确前后依赖的多阶段任务 |
| `--fanout` | 扇出模式：先并行探索再汇聚分析 | 需多角度收集信息再统一分析 |
| `--hybrid` | 混合模式：分支+汇合（默认） | 大多数复杂任务 |
| `--nodes=N` | 建议拆分为 N 个子任务 | 用户对粒度有明确要求时 |
| 无参数 | 默认 `--hybrid`，自动判断粒度 | 兼容性好 |

- **`<task-desc>`** 是 `$ARGUMENTS` 中去掉所有 `--flag` 后的剩余文本

### 3. DAG 分解思维框架

#### 3.1 通用分解步骤

```
步骤 1: 理解任务 → 识别出 3-7 个可独立分配的子任务
步骤 2: 标记依赖 → 哪些子任务需要等另一些先完成？
步骤 3: 设计拓扑 → 没有依赖的可以并行，有依赖的串成链
步骤 4: 分配智能体 → 从 ## Available Subagents 中选最合适的
步骤 5: 执行 → 调用 subagent_dag(nodes, edges)
步骤 6: 汇总 → 优先只读叶子节点结果，满意即止不上查中间节点
```

#### 3.2 常见的 DAG 拓扑模式

```
模式 A: 流水线（Pipeline）[--pipeline]
  [调研] → [设计] → [实现] → [验证]
  适用：有明确前后依赖的多阶段任务

模式 B: 扇出/汇聚（Fan-out）[--fanout]
        ┌→ [探索 A] ─┐
  [规划] ─→ [探索 B] ─→ [综合分析]
        └→ [探索 C] ─┘
  适用：需要从多个角度收集信息再统一分析

模式 C: 混合（Hybrid）[--hybrid]（默认）
        ┌→ [实现 A] ─→ [测试 A] ─┐
  [设计] ─→ [实现 B] ─→ [测试 B] ─→ [集成验证]
        └→ [实现 C] ─→ [测试 C] ─┘
  适用：多模块并行开发或多角度分析的复杂任务
```

### 4. DAG 设计原则

#### 4.1 节点粒度原则

| 粒度 | 节点数 | 适用场景 |
|------|--------|----------|
| 粗粒度 | 2-3 个节点 | 任务边界清晰，每个节点工作量较大 |
| 中粒度 | 4-6 个节点 | 大多数复杂任务（推荐） |
| 细粒度 | 7-10 个节点 | 大型多步骤任务，需精细控制 |

- **每个节点应该是逻辑上完整的子任务**，一个节点完成一个明确的产出
- **避免过度拆分**：如果一个节点只需 1-2 次工具调用就能完成，说明太细了
- **避免拆分不足**：如果一个节点要完成多个互相独立的事，说明太粗了
- **节点描述要包含完整上下文**：子智能体没有主会话的历史，描述里要写清楚：
  - 目标是什么
  - 已知信息有哪些
  - 输出格式要求是什么
  - 最终结论要能直接给下游使用

#### 4.2 依赖与并行

- **没有依赖关系的节点必须并行**——这是使用 DAG 的核心价值
- 有依赖的节点通过 `edges` 声明，系统会自动等上游完成再启动下游
- 下游节点的 context 中会自动注入上游结果，因此上游输出要结构清晰

#### 4.3 智能体选择

从 `## Available Subagents` 列表中根据每个 agent 的 description 选择最合适的。
选择原则：

| 任务类型 | 推荐 agent 特征 |
|----------|----------------|
| 信息收集、只读分析 | 选择描述为"代码探索"或"只读分析"的 agent |
| 读写改的实现任务 | 选择有写/编辑权限的通用 agent |
| 代码审查、质量检查 | 选择描述为"审查"或"检查"的 agent |
| 架构设计、方案规划 | 选择有分析能力的 agent |

### 5. 执行流程

```
1. 分析任务 → 设计 DAG 结构（确定 nodes + edges）

2. 判断是否需要用户确认：
   → 简单设计（≤3 个节点，拓扑直观）：直接执行，无需等待确认
   → 复杂设计（4+ 节点，或有多层依赖）：先用文字描述 DAG 结构让用户看一眼
     （用户没异议就直接执行，有异议则调整）

3. subagent_dag({ nodes: [...], edges: [...] })
   → 调用后系统等待所有节点执行完成，返回各节点结果与 echarts 可视化

4. 【关键】结果选择性消费：优先只读最下游节点
   → 从你定义的 edges 中，找出所有没有作为 from 出现过的节点 ID
     （即 edges 中没有任何一条边的 from 指向它）
   → 这些是叶子节点 = 最终的汇聚/产出节点，包含最重要的结论
   → 如果叶子节点结果满意 → 直接汇总输出，不再查询其他节点
   → 如果叶子节点结果不完整 → 追溯上游，补充读取关键中间结果

5. 综合分析，给出完整报告
```

⚠️ **不要一次性 subagent_status 查全部节点日志**。下游节点已经汇聚了上游的精华。
   只查叶子节点就能完成报告时，坚决不查中间节点，避免上下文被中间过程淹没。

### 6. 硬性规则：所有节点必须在同一个 DAG 中

🚨 **无论任务多复杂、拓扑有多少"层"，必须只调用一次 `subagent_dag`。**

错误做法（❌ 把多层拆成多个 DAG 串行执行）：
```
subagent_dag({ nodes: [设计] })            ← 第一次调用
subagent_dag({ nodes: [实现A, 实现B] })    ← 第二次调用（错误！）
subagent_dag({ nodes: [测试A, 测试B] })    ← 第三次调用（错误！）
```

正确做法（✅ 一次调用，所有节点 + 所有边）：
```
subagent_dag({
  nodes: [设计, 实现A, 实现B, 实现C, 测试A, 测试B, 测试C, 集成验证],
  edges: [
    { from: "设计",  to: "实现A" },
    { from: "设计",  to: "实现B" },
    { from: "设计",  to: "实现C" },
    { from: "实现A", to: "测试A" },
    { from: "实现B", to: "测试B" },
    { from: "实现C", to: "测试C" },
    { from: "测试A", to: "集成验证" },
    { from: "测试B", to: "集成验证" },
    { from: "测试C", to: "集成验证" },
  ]
})
```

**系统会自动按拓扑排序分批并行执行**：同一层的节点并行，下一层等上一层完成。
你不需要分批调用，也不需要自己管理执行顺序——交给 `subagent_dag` 内部的依赖解析。

如果你发现自己想"先调一次 DAG、等结果、再调一次 DAG"——停下来，把所有节点合并到一次调用里。

### 7. 节点 description 编写要点

```
好的 description（子智能体能直接干活的）：
  "在 src/auth/ 目录下分析登录流程，找出所有 API 端点定义，
   输出格式：每个端点一行，包含 method, path, 中间件列表"

坏的 description（子智能体看完不知道干什么的）：
  "分析认证模块"
  "看一下登录流程"
```

**描述必须包含**：
- 工作范围（哪个目录、哪些文件）
- 具体要做什么（读什么、找什么、改什么）
- 输出格式要求（让下游节点或主智能体能直接使用）

### 8. 边界

**会做**：
- 分析任务结构，设计合理的 DAG 拓扑
- 为每个节点从 Available Subagents 中选择最合适的 agent
- 在节点描述中注入足够的上下文
- 执行完成后汇总各节点结果

**不会做**：
- 不分析就直接把整个任务丢给一个 subagent
- 让节点之间存在隐含依赖但不声明 edges
- 给节点的描述太模糊（没有任何上下文信息）
- 可以并行却不并行（浪费 DAG 的价值）

### 9. 错误恢复

如果某个节点执行失败：
- **优先看叶子节点的结果**——即使上游有节点失败，如果叶子节点依然输出了可用结果，直接使用
- 如果叶子节点结果受影响，再追溯具体失败的上游节点，用 `subagent_status` 查看错误详情
- 判断是否可重试（修复描述后重新派发该节点）
- 已自动标记为 skipped 的下游节点不需要手动处理
- 决定是否整体重试或局部修复

⚠️ 不要一次性查全部节点状态。先看叶子节点，回溯式排查。

### 10. 示例

#### 示例 1：代码库分析（扇出模式）

用户执行 `/dag --fanout 分析这个项目的认证和支付模块结构`

```
nodes:
  - id: "explore_auth"
    agent: "<适合代码探索的 agent>"
    description: "探索 src/auth/ 目录，找出所有路由定义、
                  中间件、数据库模型，列出文件结构和关键函数"
  - id: "explore_payment"
    agent: "<适合代码探索的 agent>"
    description: "探索 src/payment/ 目录，找出所有 API 端点、
                  支付流程、数据库模型，列出文件结构和调用链"
  - id: "summary"
    agent: "<适合综合分析的 agent>"
    description: "综合 auth 和 payment 的探索结果，
                  输出两个模块的架构对比和关系分析"

edges:
  - from: "explore_auth"  to: "summary"
  - from: "explore_payment" to: "summary"
```

#### 示例 2：功能开发（流水线模式）

用户执行 `/dag --pipeline 给项目添加用户注册功能`

```
nodes:
  - id: "design"
    agent: "<适合代码探索的 agent>"
    description: "分析现有数据库 schema 和用户模型，
                  找出已有 auth 相关代码，输出注册功能需要修改的文件清单"
  - id: "implement"
    agent: "<适合读写改的通用 agent>"
    description: "基于 design 的输出，
                  实现用户注册功能（路由、验证、数据库操作）"
  - id: "test"
    agent: "<适合读写改的通用 agent>"
    description: "为注册功能编写测试用例，
                  覆盖正常流程和边界情况"

edges:
  - from: "design"     to: "implement"
  - from: "implement"  to: "test"
```

#### 示例 3：多模块并行重构（混合模式）

用户执行 `/dag 把项目所有 v1 API 路由迁移到 v2`

```
nodes:
  - id: "analyze"
    agent: "<适合代码探索的 agent>"
    description: "...分析现有 v1 路由定义..."
  - id: "migrate_users"
    agent: "<适合读写改的通用 agent>"
    description: "...迁移用户相关路由..."
  - id: "migrate_orders"
    agent: "<适合读写改的通用 agent>"
    description: "...迁移订单相关路由..."
  - id: "migrate_payments"
    agent: "<适合读写改的通用 agent>"
    description: "...迁移支付相关路由..."
  - id: "verify"
    agent: "<适合代码审查的 agent>"
    description: "...统一验证所有迁移后的路由..."

edges:
  - from: "analyze"          to: "migrate_users"
  - from: "analyze"          to: "migrate_orders"
  - from: "analyze"          to: "migrate_payments"
  - from: "migrate_users"    to: "verify"
  - from: "migrate_orders"   to: "verify"
  - from: "migrate_payments" to: "verify"
```
