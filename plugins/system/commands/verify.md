## 参数说明

`$ARGUMENTS` 是要验证的声明。

- **`--tests`**：验证测试通过
- **`--build`**：验证构建成功
- **`--all`**：验证所有检查
- **`$ARGUMENTS`** 其他文本作为用户声明（如"测试通过"、"bug已修复"）

无参数时，根据 `$ARGUMENTS` 推断要验证什么。

## 铁律

```
未经最新验证证据，不声称任何状态
```

如果未在当前消息中运行验证命令，不能声称通过。

## 验证门函数

```
在声称任何状态前：

1. 识别：什么命令证明此声明？
2. 运行：执行完整命令（全新、完整）
3. 读取：完整输出，检查退出码，计数失败
4. 验证：输出是否确认声明？
   - 如果否：用证据说明实际状态
   - 如果是：用证据说明声明
5. 仅那时：做出声明

跳过任何步骤 = 撒谎，不是验证
```

## 常见验证需求

| 声明 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 失败 | 上次运行、"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查、推断 |
| 构建成功 | 构建命令：退出 0 | Linter 通过、日志看起来好 |
| Bug 修复 | 测试原始症状：通过 | 代码变更、假设修复 |
| 回归测试工作 | 红-绿循环验证 | 测试通过一次 |
| Agent 完成 | VCS diff 显示变更 | Agent 报告"成功" |
| 需求满足 | 逐行检查清单 | 测试通过 |

## 红牌 - 停止

- 使用"应该"、"可能"、"看起来"
- 在验证前表达满意（"好！"、"完美！"、"完成！"等）
- 提交/推送/PR 前未验证
- 信任 Agent 成功报告
- 依赖部分验证
- 认为"就这一次"
- 疲惫且想结束
- **任何暗示成功但未运行验证的措辞**

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过了" / "看起来正确"
```

**回归测试（TDD 红-绿）：**
```
✅ 写 → 运行（通过）→ 还原修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了回归测试"（没有红-绿验证）
```

**构建：**
```
✅ [运行构建] [看到：退出 0] "构建通过"
❌ "Linter 通过了"（linter 不检查编译）
```

**Agent 委托：**
```
✅ Agent 报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 信任 Agent 报告
```

## 何时应用

**始终在：**
- 任何成功/完成声明的变化
- 任何满意表达
- 任何关于工作状态的正向陈述
- 提交、PR 创建、任务完成
- 转向下一个任务
- 委托给 Agent

**规则适用于：**
- 确切短语
- 释义和同义词
- 成功的暗示
- 任何暗示完成/正确的通信

## 底部原则

**验证没有捷径。**

运行命令。读取输出。然后声称结果。

这是不可妥协的。
