## 参数说明

`$ARGUMENTS` 是用户输入的问题描述。

## 铁律

```
未经根因调查，不进行修复
```

如果未完成第一阶段，不能提出修复方案。

## 适用场景

**任何技术问题：**
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

## 四阶段流程

### Phase 1: 根因调查

**在尝试任何修复前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切解决方案
   - 完全阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能可靠触发吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不可复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变更可能导致这个问题？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统的证据收集**

   **当系统有多个组件时（CI → 构建 → 签名、API → 服务 → 数据库）：**

   **在提出修复前，添加诊断 instrumentation：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传递
     - 检查每层状态

   运行一次以收集证据，显示哪里出了问题
   然后分析证据识别故障组件
   然后调查该特定组件
   ```

5. **追踪数据流**

   **当错误在调用栈深处时：**
   - 错误值从哪里产生？
   - 谁用错误值调用了它？
   - 一直追踪到找到源头
   - 在源头修复，而非症状

### Phase 2: 模式分析

**修复前找到模式：**

1. **找到工作的例子**
   - 在同一代码库中找到类似的代码
   - 什么有效且与坏的类似？

2. **与参考对比**
   - 如果实现模式，完全阅读参考实现
   - 不要略读 - 每行都读
   - 完全理解模式后再应用

3. **识别差异**
   - 工作和坏的之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那不重要"

4. **理解依赖**
   - 它需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做什么假设？

### Phase 3: 假设与测试

**科学方法：**

1. **形成单一假设**
   - 清楚地说明："我认为 X 是根因，因为 Y"
   - 写下来
   - 具体，而非模糊

2. **最小化测试**
   - 做最小可能的变更来测试假设
   - 一次一个变量
   - 不要同时修复多个事情

3. **验证后继续**
   - 成功了吗？ → Phase 4
   - 失败了？ → 形成新的假设
   - 不要在顶部添加更多修复

4. **当不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 请求帮助
   - 做更多研究

### Phase 4: 实现

**修复根因，而非症状：**

1. **创建失败的测试用例**
   - 最简单的复现
   - 尽可能自动化测试
   - 单次测试脚本如果没有框架
   - 修复前必须有
   - 用于编写正确失败测试的 `test-driven-development` 技能

2. **实现单一修复**
   - 解决已识别的根因
   - 一次一个变更
   - 没有"顺手改进"
   - 没有捆绑重构

3. **验证修复**
   - 测试现在通过？
   - 其他测试仍然通过？
   - 问题实际解决了？

4. **如果修复不工作**
   - 停止
   - 数数：尝试了多少修复？
   - 如果 < 3：返回 Phase 1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构**

5. **如果 3+ 修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每个修复在不同地方揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"才能实现
   - 每个修复在其他地方产生新症状

   **停止并质疑基本原则：**
   - 这个模式根本上是健全的吗？
   - 我们是在"凭惯性坚持"吗？
   - 应该重构架构还是继续修复症状？

## 红牌 - 停止并遵循流程

如果发现自己想：
- "先快速修复，稍后调查"
- "试试改 X，看看是否有效"
- "添加多个变更，运行测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复那个"
- "我不完全理解但这可能有效"
- "模式说 X 但我会不同地适应它"
- "主要问题如下：[列出修复但不调查]"
- **在已经尝试 2+ 次后"再试一次修复"**

**这些都是：停止。返回 Phase 1。**

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根因** | 读错误、复现、检查变更、收集证据 | 理解是什么和为什么 |
| **2. 模式** | 找到工作例子、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 已确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug 解决、测试通过 |

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单 bug 也很快。 |
| "紧急，没时间走流程" | 系统化调试比猜测修复更快。 |
| "先试试这个，然后调查" | 第一个修复设定模式。一开始就做对。 |
| "确认修复有效后再写测试" | 无测试的修复不牢固。测试优先才能证明。 |
| "一次修复多个节省时间" | 无法隔离什么有效。造成新 bug。 |
| "参考太长，我会适应" | 部分理解保证 bug。完全读懂再应用。 |
| "我看到问题了，让我修复" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（2+ 次后） | 3+ 次失败 = 架构问题。质疑模式，不要修复。 |
