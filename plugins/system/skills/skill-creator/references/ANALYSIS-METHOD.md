# 如何分析参考技能

分析一个优秀技能的目的是提取可复用的设计模式，而不是简单复制结构。

## 分析框架

### 1. 触发机制分析

**检查项**：
- description 字段如何写得精准？
- 触发词如何设计（具体还是模糊）？
- 技能的边界在哪里？

**分析示例**（web-video-presentation）：
```markdown
description: 把一篇文章或口播稿，做成"看起来像视频"的点击驱动 16:9 网页演示...
```
- 第一句：明确能力（做成网页演示）
- 第二句：描述方式（点击驱动）
- 触发场景：口播稿、文章、视频演示需求

### 2. 工作流架构分析

**检查项**：
- 主流程分几个阶段？
- 各阶段的输入输出是什么？
- 阶段之间的依赖关系？
- 有哪些"硬性节点"？

**分析示例**（web-video-presentation）：
```
Phase 1 内容编写
   1.1 识别用户输入
   1.2 一次产出 script.md + outline.md
        ↓
[Checkpoint Plan] ← 必须停
        ↓
Phase 2 网页开发
   2.1 脚手架
   2.2 第1章（强制anchor）
        ↓
   [硬节点] 用户验收第1章
        ↓
   2.3 第2~N章
        ↓
[Checkpoint Audio]
        ↓
Phase 3 音频合成（可选）
        ↓
Phase 4 录屏+后期
```

**关键发现**：
- Checkpoint 是强制停顿点
- 第1章是必须完整验收的锚点
- Phase 3 是可选分支

### 3. 决策树分析

**检查项**：
- 复杂场景如何分支处理？
- 条件判断的逻辑是什么？
- 有无"按能力降级"的兜底策略？

**分析示例**（自检协议）：
```
1. Agent Teams（最优）
2. subAgent（次优）
3. 自检（兜底）
```

### 4. 文件组织分析

**检查项**：
- 多个文件如何分工？
- 何时需要拆分？何时可以合并？
- 文件之间的引用关系？

**分析示例**（web-video-presentation）：
```
SKILL.md (主入口，440行)
    ↓ 引用
references/SCRIPT-STYLE.md (Phase 1.2 必读)
references/OUTLINE-FORMAT.md (Phase 1.2 必读)
references/CHAPTER-CRAFT.md (Phase 2.4 每章单一必读)
references/AUDIO.md (Phase 3 才读)
references/RECORDING.md (Phase 4 才读)
references/THEMES.md (选主题时)
```

**设计原理**：
- 按"何时读"标注，避免信息过载
- 不同阶段读不同文件
- Phase 2.4 会重复 N 次 → 单一入口文件

### 5. 约束与规范分析

**检查项**：
- 有哪些硬性规则？
- 规则如何确保执行到位？
- 违规如何处理？

**分析示例**（narrations.ts 约束）：
```markdown
narrations.ts 是 step 数和音频合成的唯一真相源。
章节 .tsx 里的 if (step === N) 出现的最大 N + 1 必须等于 narrations.length。
```

**设计原理**：
- 单一真相源防止数据漂移
- 五处地方（script/outline/章节代码/chapters.ts/音频文件）永不漂

## 分析输出模板

完成分析后，输出以下内容：

```markdown
## [技能名] 分析结论

### 1. 核心能力
[技能提供的核心价值]

### 2. 触发机制
- 触发词：[...]
- 适用场景：[...]

### 3. 工作流架构
- 阶段数：N
- 硬性节点：N 个
- 可选分支：N 个

### 4. 设计模式
- [模式1]：[描述]
- [模式2]：[描述]

### 5. 文件结构
- 总文件数：N
- 核心文件：[...]
- 按需文件：[...]

### 6. 约束机制
- 硬性规则：N 条
- 自检清单：N 项

### 7. 可复用要素
- [要素1]：适用于 [场景]
- [要素2]：适用于 [场景]
```

## 常见模式清单

| 模式类型 | 识别信号 | 复用价值 |
|---|---|---|
| 阶段+Checkpoint | 有明确的 Phase 和停顿点 | 复杂任务的流程控制 |
| 单一真相源 | 关键数据只存一处 | 多文件协作的数据一致性 |
| 按需渐进披露 | 有"何时读"标注 | 大型技能的认知负荷管理 |
| 自检协议 | 有完成后的强制检查流程 | 交付质量保障 |
| 降级策略 | 有最优/次优/兜底的分支 | 容错性和鲁棒性 |
