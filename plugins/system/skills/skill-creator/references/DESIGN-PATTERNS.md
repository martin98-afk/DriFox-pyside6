# 优秀技能的设计模式

技能设计中有一些经过验证的模式，可以直接复用。

## 模式 1：阶段 + 硬性 Checkpoint

### 核心结构
```
Phase 1 → [Checkpoint A] → Phase 2 → [Checkpoint B] → Phase 3
```

### 适用场景
- 复杂多阶段任务
- 需要用户确认才能继续的场景
- 需要防止走偏的流程

### 设计要点

1. **Checkpoint 命名**：用方括号包裹，如 `[Checkpoint Plan]`
2. **停顿原因**：明确说明为什么必须停（对齐什么）
3. **继续条件**：用户确认后继续

### 示例（web-video-presentation）
```markdown
   ▼
[Checkpoint Plan]      ← 必须停。一次对齐 5 件事：
                         稿子 / outline / 主题 / 素材 / 开发模式
   ▼
Phase 2   网页开发
```

### 复用指南
- 2~3 个 Checkpoint 最常见（太少没意义，太多阻碍流程）
- Checkpoint 放在"可能走偏"的节点
- 明确告诉用户要确认什么

## 模式 2：分层文档架构

### 核心结构
```
SKILL.md (主入口，≤100行)
├── REFERENCES/
│   ├── CONCEPT-A.md (详细概念)
│   ├── CONCEPT-B.md (详细概念)
│   └── EXAMPLES/ (示例)
└── scripts/ (工具脚本)
```

### 设计原理

| 层级 | 文件数 | 行数限制 | 用途 |
|---|---|---|---|
| 主入口 | 1 | ≤100行 | 快速理解 + 工作流 + 索引 |
| 详细参考 | 按需 | 无限制 | 深度实现时的完整指南 |
| 示例 | 可选 | 无限制 | 结构参考，不是模板 |

### 何时拆分

**应该拆分**：
- SKILL.md 超过 100 行
- 内容有明显的领域边界（如 audio/recording）
- 高级功能不常需要

**不应拆分**：
- 所有内容 < 50 行
- 内容高度相关无法分开
- 拆分后增加理解成本

### 示例（web-video-presentation）
```
SKILL.md (440行) → 拆分
├── references/CHAPTER-CRAFT.md (224行) — 每章开发必读
├── references/SCRIPT-STYLE.md (404行) — Phase 1.2 必读
├── references/OUTLINE-FORMAT.md (203行) — Phase 1.2 必读
├── references/AUDIO.md — Phase 3 才读
├── references/RECORDING.md — Phase 4 才读
└── references/THEMES.md — 选主题时读
```

## 模式 3：单一真相源

### 核心原理

关键数据只存储在一个地方，其他地方只读不写。

### 适用场景
- 多文件协作
- 需要保持数据一致性
- 有多个消费者依赖同一数据

### 示例（web-video-presentation）
```markdown
narrations.ts 是 step 数和音频合成的唯一真相源。

约束：章节 .tsx 里的 if (step === N) 出现的最大 N + 1 
必须等于 narrations.length。
```

这保证了五处地方（script/outline/章节代码/chapters.ts/音频文件）永不漂。

### 设计要点

1. **明确标注**："唯一真相源"、"禁止在此之外修改"
2. **约束规则**：定义数据格式和验证方式
3. **漂移检测**：提供自检方式发现不一致

## 模式 4：口述检查清单

### 核心原理

完成产出后必须走自检 → 修复 → 再汇报。不能跳过验证环节。

### 典型结构
```markdown
## 完工自检（完成后强制执行）

> ⚠️ 硬性流程：完成后必须走自检 → 修复 → 汇报 三步。

**执行方式**（按能力降级）：
1. Agent Teams（最优）
2. subAgent（次优）
3. 自检（兜底）

拿到结论后：先改完代码，再向用户汇报。
```

### 适用场景
- 交付物需要质量保障
- 完成和验收之间需要验证
- 有明确的质量标准可检查

### 设计要点

1. **清单要具体**：每条可验证，不是模糊的"质量好"
2. **有修复路径**：fail 了怎么办，要明确
3. **按能力分级**：确保各种环境都能执行

## 模式 5：按需渐进披露

### 核心原理

不同阶段读不同的文件，避免一次性信息过载。

### 典型结构

| 阶段 | 必读（每次都看） | 一次性读完/按需查 |
|---|---|---|
| Phase 1 | A.md + B.md | — |
| Phase 2 | C.md（每章） | D.md（一次） |
| Phase 3 | — | E.md（按需） |

### 设计要点

1. **标注"何时读"**：避免遗漏或重复
2. **阶段高频文件**：放在主入口直接引用
3. **长文本用锚点**：方便定位

### 示例（web-video-presentation）
```markdown
| 阶段 | 必读（每次都看） | 一次性读完/按需查 |
| Phase 1.1-1.2 | SCRIPT-STYLE.md + OUTLINE-FORMAT.md | — |
| Phase 2.4 | CHAPTER-CRAFT.md（每章单一入口）| EXAMPLES/ |
| Phase 3 | AUDIO.md | — |
```

## 模式 6：降级策略

### 核心原理

同一操作有多个实现路径，按能力降级确保可执行。

### 典型结构
```markdown
**执行方式**（按能力降级）：
1. Agent Teams（最优）— 需要 Teams 能力
2. subAgent（次优）— 能开子智能体
3. 自检（兜底）— 当前 agent 自己执行
```

### 设计要点

1. **明确优先级**：最优/次优/兜底
2. **条件明确**：需要什么能力
3. **结果等价**：不同路径得到相同输出

## 模式 7：约束 + 自由度平衡

### 核心原理

有些东西必须约束（换主题不破），有些东西完全放开（动画选型）。

### 典型结构
```markdown
### 必须用 token（换主题不破）
- 颜色：--shell / --surface / --text / --accent
- 字体：--font-display-cn / --font-body

### 可硬编码（按内容自由）
- 字号：80px / var(--t-h1) 都行
- 动画时长：按意图写具体值
```

### 设计要点

1. **约束底线要清晰**：什么叫"必须"
2. **自由度边界要明确**：什么叫"允许"
3. **用 token 避免硬编码**：颜色、字体等语义化

## 模式组合示例

### 组合 1：阶段+Checkpoint+自检

```markdown
Phase 1 → [Checkpoint] → Phase 2 → [硬节点] → Phase 3

自检清单（每阶段产出后强制）：
- [ ] 约束 A
- [ ] 约束 B
```

### 组合 2：分层+按需披露

```markdown
SKILL.md (主入口)
├── Phase 1 必读：A.md
├── Phase 2 每章必读：B.md
├── Phase 3 按需：C.md
└── EXAMPLES/（结构参考）
```

### 组合 3：单一真相源+约束规则

```markdown
narrations.ts = 唯一真相源

约束：
- 数组长度 = 章节代码最大 step N + 1
- 每条文本与 script.md 对应段落语义一致
```

## 模式选择指南

| 需求 | 推荐模式 |
|---|---|
| 多阶段流程 | 阶段+Checkpoint |
| 质量保障 | 口述检查清单 |
| 大型技能 | 分层文档架构 |
| 多文件协作 | 单一真相源 |
| 信息过载 | 按需渐进披露 |
| 容错性 | 降级策略 |
| 主题适配 | 约束+自由度平衡 |
